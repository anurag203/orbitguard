## Decisions Made

- Product should now open on a minimal cinematic landing route because the user wants judges and non-space users to understand OrbitGuard before entering operational tools.
- Use a custom Three.js Earth scene instead of adding a heavier globe package.
- Use Vite proxy locally and `ORBITGUARD_API_PROXY_TARGET` in Docker.
- Use Node 22 for frontend work; `.nvmrc` added.
- Promote the report panel after report generation so the briefing is visible without manual scrolling.
- Keep 3D assets procedural and deterministic so the demo works offline and screenshots remain reproducible.
- Represent local relevance visually with an ISTRAC-to-CARTOSAT tracking beam instead of adding more explanatory text.
- Use the official NASA Blue Marble texture instead of generated land shapes for Earth realism.
- Put the decision stack above raw evidence so the right rail feels like an operator workflow, not a backend dump.
- Candidate comparison and report export are part of the final-product baseline, not deferred presentation extras.
- Keep live TLE refresh explicitly non-blocking: if CelesTrak is unavailable, the UI displays the offline fallback mode rather than breaking the demo.
- Cinematic replay is an operator-controlled mode layered on scenario timeline events, not a separate landing-page sequence.
- Mission Control remains the hero operational route, but Catalog, Predictor, Closest Approach, Reports, and Learn are separate product routes instead of crowded panels on one screen.
- Closest Approach should compare scenarios before ranking the active one so judges can understand the scenario space without already knowing orbital mechanics.
- Closest Approach should behave like a tactical triage board: command summary and operator action first, scenario comparison second, sorted close-approach evidence third.
- Manual Earth controls should be layered as camera offsets, not a replacement for scenario camera fly-to, so the product remains both cinematic and operator-explorable.
- Catalog rows should be selectable operational records, not static rows, because judges need to inspect what data the system is acting on.
- Cinematic captions should stay fully opaque for recording reliability; use motion-only animation for replay captions.
- Mission Control should expose a director-level summary before the dense console panels so non-space judges immediately see the active objective and core risk numbers.
- Keep the director strip as a full-width operational band rather than another nested card; it should orient the user without shrinking the 3D stage too far.
- Add System as a concise route label for `/architecture` so technical depth is discoverable without overcrowding the top navigation.
- Keep the System route offline-safe and mostly static, while reflecting current demo/catalog/scenario status in summary metrics.
- System should prove technical depth quickly: status command band first, end-to-end pipeline second, engine modules and validation evidence third, then jumps back into operational proof routes.
- Keep the topbar status visible only on very wide screens; at laptop/projector widths the Command Deck prioritizes compact route tabs and guided demo launch.
- Use framer-motion only for frontend presentation state and route/panel choreography; backend science and API contracts remain unchanged.
- Keep Reports useful before generation by showing a draft briefing desk instead of an empty placeholder.
- Reports should feel like an operator briefing desk: draft dossier first, evidence chain status second, export-ready document with source IDs and audit columns after generation.
- Keep primary navigation limited to the judge-critical workflow; use Intel, Home launchpad, and contextual jumps for secondary education/ranking routes.
- Treat the Home route as a cinematic entry and explanation surface, not a dense dashboard; deeper operational detail belongs in Mission Control, Catalog, Predictor, Reports, and System.
- On Home, let HUD overlays pass pointer input through to the WebGL canvas except for explicit CTA/loadout controls, so judge demos can still drag/zoom the Earth without UI layers blocking the gesture.
- Mission Control should prioritize the Earth stage and command bar over side-panel density; side rails remain useful, but the 3D stage is the main judge-facing surface.
- Catalog should prioritize immediate evidence inspection: selected object details and orbit preview belong above summary/safety prose so judges see source data without hunting.
- Predictor should tell a simulation story before exposing the candidate list: target, current risk, burn scan, post-burn risk, and recommended maneuver are the first-read path.
- Learn should behave like a compact training simulator, not a static glossary, so non-space judges can learn the minimum concepts and immediately jump into Catalog, Closest Approach, and Predictor.
- Guided Demo should be presenter-controlled, not timer-controlled, so recording and live judging stay synchronized with the actual route being shown.
- Home should feel like a production website, not a dashboard: full-bleed Earth background first, focused mission promise second, compact proof metrics third, and workflow modules below.
- On Home desktop, hero layout containers should pass pointer events through empty space so the rotating Earth remains interactive; explicit CTAs/cards keep pointer events.
- Global navigation should stay opinionated: the desktop deck shows Mission, Catalog, Predictor, Briefing, and System only; Risk Board and Learn remain available through Intel and mobile drawer so the topbar feels like a production app instead of a complete sitemap.
- The Predictor route remains `/avoidance`; visible copy can say Predictor/Burn planner, but accessible labels should preserve "Avoidance" so screen-reader users and tests can find the avoidance-planning workflow.
- Mission route should behave like the first operational cockpit, not a second landing page: keep the hero compact, show the Earth stage early, keep the current decision drawer adjacent to the visual scene, and reveal evidence as action rhythm rather than raw telemetry.
- Catalog route should read as source evidence and object triage, not a table: source status and object-class rows first, rich inspector second, raw TLE only behind disclosure.
- Predictor should present burn planning as an operator simulation: target lock and visual scene first, risk-collapse board second, candidate cards third, raw assumptions behind disclosure.
- Reports should behave like the audit desk: draft outline first, generated mission dossier second, evidence chain/source IDs/export state always visible, and assumptions behind disclosure.
- System should behave like a technical command center, not a static architecture note: release posture first, pipeline and validation second, engine contracts third, and limitations/evidence jumps last.
- System pipeline stages should be selectable controls with a focused inspector; route jumps should live in the inspector as proof actions so judges learn the module contract before leaving the technical view.
- Mission Director should be a presenter layer, not another data route: keep scene count fixed, use operator-friendly labels, keep cue navigation manual, and never force judges through a timer.
- Mission Sync should stay concise and route-adjacent: show human mission state and stage progress, not raw store values or backend response payloads.
- Catalog empty states must stay operator-grade: no `n/a`, `unknown`, or loading tags should appear when filters return no objects.
- Catalog source and metadata fallbacks should use evidence language: source sync, fixture handshake, owner unlabelled, not supplied. Do not show raw loading, unknown, or n-a labels in the Catalog browser or inspector.
- Avoidance and Risk pre-data states must be designed operator states, not backend placeholders: use Simulation pre-flight, queued candidate slots, sync placeholders, and action-oriented copy instead of Loading/Pending/Waiting labels.
- Avoidance should show the maneuver as a staged operator workflow, not just metrics: Lock geometry, Scan burns, Select burn, Screen secondary, and Build brief remain visible and inspectable while plan/apply/report state changes.
- Mission pre-plan state should show an operator readiness briefing before phase details; never show raw Loading/Awaiting labels once the route is settled.
- Reports pre-generation state should read as an intentional briefing queue, not a draft/pending backend state; planner, safety, source, and burn-scan inputs may be queued, but raw pending/not-planned labels should not reach the UI.
- System readiness state should never show loading, booting, or ellipsis placeholders. Before demo status arrives, show readiness sync, catalog sync, scenario sync, and checks arming; after data arrives, show explicit ready/attention metrics.
- Home and app-shell readiness states should never show booting/loading placeholders. Before readiness arrives, use readiness sync, source sync, mission sync, or scenario sync; keep cockpit readouts non-wrapping and keep Home Earth controls clear of telemetry cards.
- Learn should behave like a small simulator, not static docs: four training stages remain visible, one stage is actively selected, the active lesson explains what judges should notice, and the route action jumps to the proof surface.
- Reports should expose review modes instead of treating briefing content as one long dossier: Summary, Evidence, Assumptions, and Export each get a deliberate tab, and the mobile layout uses a two-by-two command pad to avoid cramped labels.
- Reports should also expose a judge-facing story path before the dossier: Situation, Decision, Safety proof, and Submission chapters stay selectable so presenters can explain the report without scrolling through raw internals.
- Catalog should lead with curated mission lenses before free-form filters, so non-space judges can immediately switch between all objects, protected ISRO assets, debris threats, and the LEO corridor without parsing raw catalog data.
- Risk Board should be a triage workflow, not a static metric board. Severity lanes are selectable controls, and empty lanes must state that they are clear rather than borrowing evidence from another lane.
